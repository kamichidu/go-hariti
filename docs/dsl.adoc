= Hariti DSL Design Specification
:toc: left
:sectnums:

This document defines the architectural design, syntax rules, Abstract Syntax Tree (AST) contracts, and compilation boundary policies for the Hariti Configuration Domain Specific Language (DSL).

== Purpose

The Hariti DSL serves strictly as an input configuration format, not the internal execution model. It enforces the repository's foundational principle: "Separation of Input Format and Internal Model" (`Input Format != Internal Model`).

The DSL is processed through a structured pipeline:
1. The DSL text is parsed into a syntactical representation called the **DSL AST**.
2. The DSL AST is compiled and transformed into the semantic representation called the **Graph IR**.

This separation ensures that compiler-level frontends can evolve independently from core graph-processing engine features.

---

== Supported Syntax

The initial implementation of the Hariti DSL supports the following syntax structures:

=== use
Declares a plugin bundle to be managed by `hariti`. Supports both a **braced block form** and an **unbraced clause form**.

==== Unbraced Clause Form
Options are written directly under the `use` directive without braces, extending until the next top-level directive (such as `use`, `include`, `replace`, `merge`) or the end of the file.
[source,hariti]
----
use Shougo/vimproc.vim
  source ./forks/vimproc
  as vimproc
  depends (
    foo/bar
  )
----

==== Braced Block Form
Options are written inside a curly braced block (`{ ... }`).
[source,hariti]
----
use Shougo/vimproc.vim {
  source ./forks/vimproc
  as vimproc
  depends (
    foo/bar
  )
}
----

=== as
Registers human-readable alias names for the associated bundle.
[source,hariti]
----
use Shougo/vimproc.vim
  as vimproc
----

=== depends
Lists directional dependencies between bundles in a parenthesized block.
[source,hariti]
----
use osyo-manga/vim-watchdogs
  depends (
    thinca/vim-quickrun
    Shougo/vimproc.vim
  )
----

=== enable_if
Specifies conditional expression strings evaluated at Vim startup to dynamically control loading.
[source,hariti]
----
use godlygeek/csapprox
  enable_if "!has('gui_running')"
----

=== build (with on <os> and on *)
Defines OS-specific commands executed post-deployment. OS values can target individual operating systems (`windows`, `mac`, `linux`) or match all environments (`*`).
[source,hariti]
----
use Shougo/vimproc.vim
  build {
    on windows
      - mingw32-make -f make_mingw64.mak
    on mac
      - make -f make_mac.mak
    on linux
      - make -f make_unix.mak
    on *
      - echo all
  }
----

=== include
Includes another `.hariti` file and merges its declarations into the current compilation unit. Paths can be unquoted, or quoted using either double quotes (`"`) or single quotes (`'`). Paths may also contain wildcard glob patterns (such as `*`) to include multiple files at once. Throughout the Hariti DSL, string literals and paths must be enclosed in either double quotes (`"`) or single quotes (`'`).

[source,hariti]
----
include plugins/core.hariti
include "plugins/ui.hariti"
include 'plugins/extra.hariti'
include plugins/*.hariti
----

`include` is a frontend-only file expansion mechanism.

It does not introduce scope, namespace, profile, or dependency semantics.
Included files are parsed and merged before AST-to-Graph transformation.

Key rules:
* **Aggregation Only**: `include` is solely used to aggregate multiple files. It does not represent package dependencies.
* **Relative Path Resolution**: Paths specified in `include` directives are resolved relative to the directory of the file containing the directive.
* **Quoting Styles**: Supports both single quotes (`'`) and double quotes (`"`). Across the entire DSL, quote marks are either `'` or `"`.
* **Glob Expansion**: Supports wildcard pattern matching using `*` (and other glob syntax) to match and include multiple files recursively relative to the inclusion context. Glob expansion is executed entirely during the include processing phase inside the DSL frontend.
* **Circular Detection**: The compiler/loader must perform circular dependency checks to prevent infinite compilation loops.

== Source Resolution

Hariti separates bundle identity from implementation location.

`use` declares the canonical bundle identity.
`source` declares where the bundle implementation is obtained from.

When `source` is omitted, the bundle ID is used as the source expression.

[source,hariti]
----
use Shougo/vimproc.vim

use my/local-plugin {
  source ./plugins/local-plugin
}
----

=== Source Classification Rules

[cols="1,2,3", options="header"]
|===
| Kind | Pattern | Meaning

| Local path
| `/`, `./`, `../`, `~`, `$VAR`, `${VAR}`
| Local filesystem path

| Explicit remote URL
| `http://`, `https://`, `ssh://`
| Remote repository URL used as-is

| Git SSH shorthand
| `git@...`
| Remote Git SSH source used as-is

| GitHub shorthand
| `owner/repo`
| Resolved to `https://github.com/owner/repo`

| Vim.org shorthand
| `name`
| Resolved to `https://github.com/vim-scripts/name`
|===

Environment variable expansion is supported only for local-path sources.
A source using environment variables must start with `$VAR` or `${VAR}`.

Unsupported ambiguous form:

[source,hariti]
----
source owner/$REPO
----

  Used for locking specific commit revisions or tags.

== AST Design

The AST types represent the direct syntactical structure of the DSL. They are located inside `internal/config/dsl/ast` and consist of the following models:

=== File
The root container representing a compiled DSL configuration file.

[cols="1,2,3", options="header"]
|===
| Field Name | Type | Responsibility
| `Bundles` | `[]BundleDecl` | Lists all parsed plugin bundle declarations inside the file.
|===

=== BundleDecl
An individual bundle declaration and its syntax clauses.

[cols="1,2,3", options="header"]
|===
| Field Name | Type | Responsibility
| `Use` | `string` | The declared plugin identifier or VCS name (e.g. `owner/repo`).
| `Aliases` | `[]string` | Registered alias strings mapped via `as` clauses.
| `Depends` | `[]string` | Names of other plugin dependencies parsed from `depends` lists.
| `EnableIf` | `string` | Condition expression string extracted from `enable_if` clauses.
| `Build` | `[]BuildBlock` | Build rules parsed from the `build` block.
|===

=== BuildBlock
An OS-specific block containing post-deployment commands.

[cols="1,2,3", options="header"]
|===
| Field Name | Type | Responsibility
| `OS` | `string` | Target operating system string (e.g. `linux`, `windows`, `mac`, `*`).
| `Commands` | `[]string` | Sequential list of commands prefix-marked with `-` under this block.
|===

== Replace and Merge

`replace` and `merge` are graph transformation directives.

They are parsed as DSL declarations, stored in the intermediate compile graph, and applied before the graph is passed to backend components.

Backend components such as sync, generation, lock, and projection must receive a resolved graph that does not contain `replace` or `merge` directives.

=== replace

`replace` fully replaces the bundle configuration while preserving the canonical bundle identity.

[source,hariti]
----
replace Shougo/vimproc.vim {
  source ./forks/vimproc
  depends (
    foo/bar
  )
  build {
    on linux
      - make
  }
}
----

Rules:

* The target ID must already exist in the compile graph.
* Fields written inside `replace` become the new bundle definition.
* Fields omitted from `replace` are cleared, not preserved.
* The canonical bundle ID remains the replace target ID.
* `replace` is useful when the implementation and dependency set should be completely overridden.

=== merge

`merge` partially overrides the target bundle declaration.

Only fields explicitly written inside the `merge` block are applied.
Fields not written are preserved from the original bundle.

[source,hariti]
----
merge Shougo/vimproc.vim {
  source ~/src/vimproc
}
----

[source,hariti]
----
merge osyo-manga/vim-watchdogs {
  depends (
    foo/bar
    bar/baz
  )
}
----

Rules:

* The target ID must already exist in the compile graph.
* Written fields overwrite the corresponding original fields.
* Omitted fields keep their original values.
* `depends (...)` inside `merge` replaces the entire dependency list.
* `build { ... }` inside `merge` replaces the entire build step list.
* `enable_if` inside `merge` replaces the original enable condition.
* `source` inside `merge` replaces the original source.
* The canonical bundle ID remains the merge target ID.

=== JSON Serialization Rule
Since `replace` and `merge` are compile-time transformation directives, they must never leak into the Resolved Graph or its serialized outputs. 
The JSON serialization of the Resolved Graph (e.g., printed by the `dump-graph` command) must completely exclude `replaces` and `merges` fields, ensuring that Graph consumers receive only the resolved plugin configurations.

== Boundary Rules

To prevent responsibility leaks and preserve long-term maintainability, the following boundary constraints are strictly enforced:

* **Syntax-Only Parser**: The `dsl` parser's single responsibility is mapping DSL text into a `dsl.ast` representation (`Parser responsibility = Syntax -> AST`).
* **Zero Semantic Dependencies**: The parser package and its generated structures must never import `internal/graph`. The parser is entirely agnostic of core execution meanings.
* **Separation of Concerns**: The AST represents "How it is declared" (syntax), whereas the Graph IR represents "What it means" (semantics).
* **Graph-Level Safeguards**: Graph semantic normalization and validation are the responsibility of the Graph package, and are triggered only *after* transforming the AST into a Graph representation.
* **Include Handling**: Include file expansion, relative path resolution, and circular dependency checks are handled entirely within the DSL frontend layer. The Graph IR does not represent `include` directives.

---

== Data Transformation Flow

Hariti configurations are compiled and processed through the following sequential pipeline stages:

[source,text]
----
DSL text
  -> Parser
  -> DSL AST
  -> Compile Graph
  -> apply replace / merge
  -> Resolved Graph
  -> Normalize
  -> Validate
  -> Backend
----

`replace` and `merge` must not leak into backend processing.

Graph consumers must not inspect replacement directives.

---

== Non-Goals

The following activities must never be performed within the Parser layer:

Dependency Resolution::
  The parser does not analyze, sort, or resolve bundle dependencies.
Replace Overrides::
  The parser does not evaluate or apply `replace` overrides.
VCS and Git Operations::
  The parser does not communicate with the file system, network, or Git commands.
RuntimePath Generation::
  The parser does not generate Vim script runtime path projection files.
