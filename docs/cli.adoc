= CLI Design Policy

== Purpose
Hariti uses the Go standard `flag` package and keeps CLI behavior small, predictable, and independent from any external CLI framework.

The CLI layer is a presentation adapter.
Core behavior must remain in the root `hariti` package.

== Package Layout
[source,text]
----
internal/cli/
  app.go
  command.go
  context.go
  flags.go
  assets/
    hariti.txt

  commands/
    install.go
    sync.go
    deploy.go
    dump_graph.go
    assets/
      install.txt
      sync.txt
      deploy.txt
      dump_graph.txt
----

== Global Flags
Global flags must support both short and long forms.

Examples:
[source,text]
----
-c, --config
-v, --verbose
----

Short and long forms must bind to the same flag variable.

In addition, the following global directory flags are supported:
[source,text]
----
--config-dir  Directory where configurations (bundles.hariti) are located
--data-dir    Directory where data (repos, metadata, generations) are located
----

== Flag Position Policy
Global flags must be accepted both before and after the subcommand.

These must be equivalent:
[source,text]
----
hariti --config-dir . install
hariti install --config-dir .
hariti install -c bundles.hariti
----

If the same global flag is specified both before and after the subcommand, the later/subcommand-local value wins.

Example:
[source,text]
----
hariti --config-dir A install --config-dir B
----

Result:
[source,text]
----
directory = B
----

== CLI Context
`internal/cli` owns a dedicated CLI context type.

It must be based on `context.Context` and carry parsed CLI concerns such as:

[source,go]
----
type Context struct {
    context.Context

    Global *GlobalFlags
    Logger hariti.Logger
    Stdout io.Writer
    Stderr io.Writer
}
----

The root CLI parser parses global flags before the subcommand, then passes the remaining arguments to the selected command.

Each subcommand parses its own arguments and re-registers global flags so that global flags are also accepted after the subcommand.

== Command Model
Subcommands must be represented as command structs, not loose `RunXXX` functions.

[source,go]
----
type Command interface {
    Name() string
    Run(ctx *Context, args []string) error
}
----

Each command owns:
* name
* usage text
* local flags
* argument parsing
* execution

=== Subcommand Self-Registration

Subcommands are registered by init-time self registration from `internal/cli/commands`.

The binary entrypoint (`cmd/hariti/main.go`) imports the commands package for registration side effects.

The root CLI only dispatches commands.

== Usage Text
The Go standard flag package is used only for parsing.

User-facing usage text is maintained as embedded text assets.

Hariti does not use flag.PrintDefaults() for help output because short and long aliases are modeled as one user-facing option.

Root CLI usage belongs to:
[source,text]
----
internal/cli/assets/hariti.txt
----

Subcommand usage belongs to:
[source,text]
----
internal/cli/commands/assets/<command>.txt
----

The built-in `flag` generated usage is not the primary user-facing help text.

== Responsibility Boundary
[source,text]
----
cmd/hariti
  = binary entrypoint only

internal/cli
  = CLI presentation layer

internal/cli/commands
  = subcommand presentation logic

package hariti
  = public usecase API

internal/*
  = implementation details
----

The CLI layer must call the root `hariti` package and must not deeply orchestrate internal implementation packages.
